# [MVP 핵심 개발 기획안] 시설 관리 시스템 어드민 (Phase 1)

## 1. 프로젝트 개요

- **목적:** 여러 업체가 가입하여 독립적으로 건물과 시설 정보를 등록하고 관리할 수 있는 초경량 멀티 테넌트(Multi-tenant) 시설 관리 어드민 뼈대 구축.
- **핵심 아키텍처:**
  - 하나의 업체(마스터 계정)는 여러 개의 워크스페이스(건물/장소)를 관리할 수 있음.
  - 모든 데이터(시설 타입, 시설 정보)는 워크스페이스 단위로 철저히 격리되며, 타 워크스페이스와 공유되거나 노출되지 않음.

## 2. 기술 스택 (Tech Stack)

- **Framework:** Next.js (App Router 기반 풀스택)
- **Database:** Supabase (PostgreSQL)
- **Authentication:** Auth.js (구 NextAuth.js)
- **Styling:** Tailwind CSS (유틸리티 퍼스트 스타일링)
- **UI Components:** shadcn/ui (Radix Primitives 기반 커스텀 컴포넌트 라이브러리)

---

## 3. 회원가입 및 관리자 설정 (Admin Onboarding)

시스템을 사용하는 관리 업체의 마스터 계정을 생성하고 로그인 세션을 유지하는 단계입니다.

- **회원가입 기본 정보:** 업체명(회사명), 관리자 이름, 전화번호, 이메일, 비밀번호
- **전화번호 입력 규칙:** 숫자 입력 시 자동으로 하이픈을 삽입하여 표시. 서울 지역번호(`02-XXXX-XXXX`)와 그 외(`0XX-XXXX-XXXX`) 형식을 구분 적용. 내부 저장은 하이픈 없는 숫자만(`01012345678`).
- **인증 인프라:** Auth.js를 이용한 관리자 로그인 세션 관리 및 보안 적용.
- **UI 구현:** shadcn/ui의 `Card`, `Form`, `Input`, `Button` 컴포넌트를 활용하여 깔끔하고 모던한 회원가입/로그인 레이아웃 구성.

## 4. 백엔드 데이터 구조 및 매핑 규칙 (Database Schema Logic)

모든 테이블의 기본 키(PK)는 변하지 않는 **순수 랜덤 고유 ID(UUID)**를 사용하여 상호 간의 유연성과 확장성을 보장합니다.

### ① 워크스페이스 (Workspace) 관리

- **개념:** 업체가 관리하는 독립적인 '건물' 또는 '장소'의 단위 (예: 샘플타워).
- **등록 및 내부 저장 필드:**
  - `workspace_name` (VARCHAR): 워크스페이스 이름
  - `max_floor` (INT): 지상 최고 층수 (예: 지상 10층인 경우 `10` 저장 / 지상 층이 없으면 `0`)
  - `min_floor` (INT): 지하 최저 층수 (예: 지하 2층인 경우 `-2` 저장 / 지하 층이 없으면 `0`)
- **층수 데이터 처리 및 치환 규칙 (중요):**
  - DB 용량 최적화 및 정렬을 위해 층수는 내부적으로 **정수(Integer)**로만 관리합니다.
  - 프론트엔드 화면에 노출할 때만 아래 규칙에 따라 문자열(String)로 치환하여 드롭다운 리스트를 자동 생성합니다.
    - **양수 (예: `3`) ➡️ 뒤에 'F' 결합 (`3F`)**
    - **음수 (예: `-1`) ➡️ '-'를 'B'로 치환 (`B1`)**
    - _예시: `max_floor: 3`, `min_floor: -1` 일 때 UI 드롭다운 생성 결과 ➡️ `['3F', '2F', '1F', 'B1']`_

### ② 데이터 격리 및 매핑 규칙

- **시설 타입 (Facility Type):**
  - 관리자가 해당 워크스페이스 내에서 사용할 공간 카테고리 메뉴 (예: 화장실, 회의실, 관리사무소, 헬스장 등).
  - **필수 매핑:** 생성된 `워크스페이스 ID`와 외래키(FK)로 연결되어야 함.
- **시설 정보 (Facility Info):**
  - 실제 관리 대상이 되는 공간의 구체적인 정보.
  - **필수 매핑:** 생성된 `워크스페이스 ID` 및 `시설 타입 ID`와 외래키(FK)로 연결되어야 함.
  - **보유 필드:**
    - **시설명:** 프리 텍스트(Free Text) 필드. 관리자가 현장에서 부르는 별칭이나 세부 구분을 자유롭게 입력 _(예: 타투인(대회의실), 3층 남자 화장실, 방제센터 등)_
    - **층 수 (INT):** 해당 워크스페이스 범위 내에서 선택된 층수의 **정수 값**을 저장 _(예: 3층은 `3`, 지하 1층은 `-1`로 매핑되어 저장)_
    - **시설 타입:** 기등록된 타입 중 선택
    - **위치 설명:** (예: 엘리베이터 좌측 복도 끝, 탕비실 안쪽 등)
    - **비고:** 특이사항 기록
- **💥 데이터 격리 조건:** A 워크스페이스에서 등록한 시설 타입, 층수 범위, 시설 정보는 B 워크스페이스에 절대 노출되거나 공유되지 않음.

### ③ 점검자 (Inspector) 관리

- **개념:** 시설 점검을 수행하는 담당자. 워크스페이스 단위로 관리되며, 동일 테넌트라도 워크스페이스별 점검자 목록이 독립적으로 구성됨.
- **필수 매핑:** `워크스페이스 ID` + `테넌트 ID` 이중 FK.
- **보유 필드:**
  - `name` (VARCHAR 100): 점검자 이름 (필수)
  - `phone` (VARCHAR 11): 전화번호, 숫자만 저장 (필수). 화면 출력 시 하이픈 포맷 적용.
  - `email` (VARCHAR 255): 이메일 (선택)
- **데이터 격리:** 워크스페이스 레벨 RLS 정책 적용 (tenant_id + workspace_id 이중 필터).

### ④ 점검표 (Checklist) 관리

- **개념:** 시설 점검 시 사용하는 체크리스트 템플릿. 워크스페이스 단위로 관리.
- **필수 매핑:** `워크스페이스 ID` + `테넌트 ID` 이중 FK.
- **보유 필드:**
  - `checklist_name` (VARCHAR 255): 점검표명 (필수)
  - `description` (TEXT): 설명 (선택)
  - `repeat_cycle` (ENUM): 반복 주기 — `daily`(매일) / `weekly`(매주) / `monthly`(매월)
  - `count` (INT): 주기당 점검 횟수 (최소 1)
  - `days` (INT[]): 요일 배열 (0=일~6=토). `repeat_cycle = 'weekly'`일 때만 사용.

### ⑤ 점검 항목 (Checklist Item)

- **개념:** 점검표를 구성하는 개별 항목. 점검표에 종속(CASCADE 삭제).
- **필수 매핑:** `checklist_id` FK.
- **보유 필드:**
  - `item_name` (VARCHAR 255): 항목명 (필수)
  - `response_type` (ENUM): 응답 방식 — 현재 `checklist`(체크리스트)만 지원
  - `is_required` (BOOLEAN): 점검자가 반드시 응답해야 하는지 여부. 기본값 `FALSE`.
  - `sort_order` (INT): 항목 표시 순서 (0부터 시작, 등록 순서대로 자동 부여)
- **UI 정책:** 항목 추가/삭제는 점검표 폼 내에서 인라인으로 처리. 점검표 저장 시 기존 항목 전체 삭제 후 재삽입(MVP 단순화).

### ⑥ 시설-점검표 연결 (Facility Checklist)

- **개념:** 시설(Facility)과 점검표(Checklist)의 M:N 관계. 하나의 시설에 여러 점검표를 지정할 수 있고, 하나의 점검표가 여러 시설에 사용될 수 있음.
- **연결 테이블:** `facility_checklists`
  - `facility_id` FK → `facilities(id)` CASCADE
  - `checklist_id` FK → `checklists(id)` CASCADE
  - `workspace_id`, `tenant_id`: 데이터 격리 보장용
  - UNIQUE `(facility_id, checklist_id)`: 중복 연결 방지
- **저장 정책:** 시설 등록/수정 시 점검표 연결 목록을 전체 교체(삭제 후 재삽입).

## 5. 어드민(Admin) 주요 기능 및 UI 동선

### ① 워크스페이스 생성 및 층수 정의 (초기 세팅)

- [워크스페이스 추가] 버튼 클릭 시 shadcn/ui의 `Dialog`(모달) 또는 전용 폼 화면 활성화.
- '워크스페이스 이름'과 함께 **'지상 층수(0~최대)'**, **'지하 층수(0~최대)'**를 숫자로 입력받음.
  - _화면 예시: "지상 [ 10 ] 층 / 지하 [ 2 ] 층"_
  - 백엔드 저장 시 지상은 그대로 `10`, 지하는 음수 형태인 `-2`로 계산되어 테이블에 인서트됨.

### ② 시설 타입 관리 메뉴 (독립 메뉴)

- 워크스페이스 대시보드 내 별도 메뉴로 분리되어 관리됨.
- 해당 장소에서 사용할 공간 카테고리를 자유롭게 추가, 수정, 삭제(CRUD) 가능.

### ③ 시설 정보 등록 및 관리 메뉴

- **등록 기능:**
  - 관리자가 [시설 등록] 버튼을 누르면 입력 폼(Form)이 활성화됨.
  - **층 수 선택 및 시설 타입 필드**는 시스템이 백엔드 데이터를 읽어와 치환 규칙(`3F`, `B1` 등)을 적용한 뒤, shadcn/ui의 `Select` 컴포넌트를 사용하여 세련된 드롭다운 형태로 노출함. 이를 통해 관리자의 오입력을 차단함.
  - 관리자는 **시설명(자유 입력), 층 수(선택), 시설 타입(선택), 위치 설명, 비고** 항목을 입력하여 등록을 진행함.
  - **점검표 연결 (선택):** 해당 워크스페이스에 등록된 점검표 목록을 체크박스로 표시하여 복수 선택 가능. 선택하지 않아도 등록 가능.
  - **사전 조건 검사:** 등록 가능한 시설 타입이 없으면 [시설 등록] 버튼 비활성화 및 안내 메시지 표시.
- **조회 기능:**
  - 등록이 완료된 시설 정보 리스트를 shadcn/ui의 `Table` 컴포넌트와 Tailwind CSS의 그리드 시스템을 활용하여 웹/태블릿 환경에 깔끔하게 최적화된 표 형태로 화면에 출력함.
  - DB에 층수가 정수(`3`, `2`, `1`, `-1`)로 저장되어 있으므로, **층수별 오름차순/내림차순 정렬 및 '3층 시설만 보기' 같은 필터링 기능을 추가적인 연산 없이 SQL 쿼리만으로 완벽하게 지원**할 수 있음.
  - 테이블에 연결된 점검표 수(N개)를 함께 표시.

### ④ 점검자 관리 메뉴

- **위치:** 워크스페이스 대시보드 내 별도 메뉴 (`/dashboard/[workspaceId]/inspectors`).
- **기능:** 점검자 추가, 수정, 삭제(CRUD). 삭제 시 재확인 다이얼로그 표시.
- **입력 항목:** 이름(필수), 전화번호(필수 — 하이픈 자동 포맷), 이메일(선택).
- **전화번호 유효성:** 숫자만, 최대 11자리. 입력 화면은 하이픈 포함 표시, 저장 시 숫자만 저장.
- **데이터 범위:** 현재 워크스페이스에 속한 점검자만 조회/관리.

### ⑤ 점검표 관리 메뉴

- **위치:** 워크스페이스 대시보드 내 별도 메뉴 (`/dashboard/[workspaceId]/checklists`).
- **기능:** 점검표 추가, 수정, 삭제(CRUD). 삭제 시 재확인 다이얼로그 표시(하위 항목도 CASCADE 삭제됨을 안내).
- **입력 항목:**
  - 점검표명(필수), 설명(선택)
  - 반복 주기: 매일 / 매주 / 매월 선택
  - 횟수: 주기당 점검 횟수 (숫자 입력, 최소 1)
  - 요일: `반복 주기 = 매주`일 때만 표시. 일~토 체크박스.
  - 점검 항목: 항목명 + 응답 방식(체크리스트 고정) 입력 행을 동적으로 추가/삭제. 최소 1개 이상 필수.
- **항목 저장 방식:** 점검표 저장 시 기존 항목 전체 삭제 후 순서대로 재삽입 (MVP 단순화 정책).
